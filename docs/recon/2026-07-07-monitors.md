## 1. ★S1/S2 — Target-host parameterizability

**ANSWER (ground truth): NOT cleanly parameterizable.** For a browser monitor the target
host is **baked into the flow code** in the spec body (the `page.goto(...)` literal), not a
declarative field. The manifest carries a `target` field but **nothing reads it** — it is
documentation, not configuration. There is **no** endpoint/HTTP monitor to compare against;
that monitor class does not exist in this repo and is not even representable in the schema.

### 1a. There is no ENDPOINT/HTTP monitor (the second half is vacuous)

OBSERVED — every manifest entry is `kind: "browser"` (16/16), and the schema's `kind` is a
single-value enum:

```
manifest.schema.json:32   "kind": { "type": "string", "enum": ["browser"] },
$ grep -c '"kind": "browser"' manifest.json  → 16   (every monitor)
```

So "a representative ENDPOINT/HTTP monitor" cannot be shown — the fleet is 100% browser
journeys. Any HTTP-check parameterizability question is answered only by whatever the
runner does for non-browser checks, which is **out of this repo** (INFERRED-unknown).

### 1b. The browser host is hardcoded per-spec in `page.goto()`

OBSERVED — every spec navigates to an absolute, hardcoded origin. Representative
(`monitors/wegmans/homepage-load.spec.ts:21`):

```ts
await page.goto('https://www.wegmans.com', { waitUntil: 'domcontentloaded' });
```

### 1c. The manifest `target` field is read by nothing

OBSERVED — `target` exists per-entry (e.g. `manifest.json:14 "target": "https://www.wegmans.com"`),
but:

- No spec references it. `grep -rniE 'process\.env|baseURL|import\.meta|env\.' monitors/`
  → **(none)**.
- `playwright.config.ts` sets **no `baseURL`** (so a relative `goto` has nothing to resolve
  against; the absolute literals are load-bearing). Config header itself says the runner
  "applies its OWN execution config" (`playwright.config.ts:5-11`).
- The manifest validator never cross-checks `target` against the spec's `goto` — corroborated
  by the (unmerged) prior deep-review §6: *"`target` … validated by **nothing** — not even
  cross-checked against the URL the spec drives"* (see §4).

So `target` is decorative: changing it would repoint **nothing**; the browser still goes
wherever the `goto` literal says.

### 1d. Verdict + lift

**Can the same artifact be pointed at a different environment's host without editing flow
code — via a spec field, env var, or runtime override? → NO.** There is no such seam today:
no `baseURL`, no `process.env`, no consumption of `target`. Repointing requires **editing the
spec source**.

Lift to make the fleet env-repointable (OBSERVED counts of host literals in *executable*
code, comments excluded):

- **16 navigation-host literals** — one `goto` (or `ENTRY_URL`) per spec, all 16 specs.
- **6 additional backend/third-party host literals** in module-level network-anchor
  regexes that also pin a host:
  - `wegapi.azure-api.net` — `meals2go-browse-menu.spec.ts:30` (`MENUS_API`),
    `meals2go-catering-browse.spec.ts:41` (`CATERING_MENUS_API`),
    `meals2go-homepage.spec.ts:38` (`BOOTSTRAP_API`)
  - `algolia.net` — `search-autocomplete.spec.ts:27` (`ALGOLIA_QUERIES`)
  - `opentable.com` — `amore-reservations.spec.ts:26` + `nextdoor-reservations.spec.ts:32`
    (`OPENTABLE_LOADER`; a fixed third party — would NOT change per Wegmans environment)

Net edit surface for a full env repoint: **~22 host literals across 16 files**, none behind a
shared declarative field. A first cut (nav hosts only) is 16 edits in 16 files; the
network-gated specs additionally need their anchor host swapped or they would gate on the
wrong environment's backend.

**Caveat / falsifier (run):** falsifier was "some spec reads an env var or `target`" —
`grep -rniE 'process\.env|baseURL|import\.meta|env\.' monitors/` returned nothing, and
`playwright.config.ts` has no `baseURL`. The seam genuinely does not exist. INFERRED-only
piece: how the *runner* injects config is out of this repo — if the runner string-replaced
hosts at sync time it could parameterize externally, but nothing in-repo exposes or expects
that.

## 2. ★S2 — Environment-derivability

**ANSWER (ground truth): every host in the fleet is a PRODUCTION host — there are zero
dev/staging exemplars — so a host→environment map can only ever confirm `prod`, and three
host shapes would be misclassified or unmatched by a naive regex and MUST render
`environment:unknown` rather than defaulting to prod.**

### 2a. Every distinct target host (grep of the real hosts)

OBSERVED. **Navigation-target origins** (what a spec's `page.goto` drives), with monitor
count:

| # | host | monitors | env (as shipped) |
|---|---|---|---|
| 1 | `www.wegmans.com` | 7 (homepage-load, recipe-nav, recipe-search, search-product, search-autocomplete, shop-category-browse, store-locator) | prod |
| 2 | `www.meals2go.com` | 4 (meals2go-homepage, meals2go-cheese-pizza-cart, meals2go-browse-menu, meals2go-catering-browse) | prod |
| 3 | `wegmansamore.com` | 2 (amore-menu, amore-reservations) — **no `www.`** | prod |
| 4 | `www.wegmansnextdoor.com` | 2 (nextdoor-homepage, nextdoor-reservations) | prod |
| 5 | `synthwatch-dashboard.vercel.app` | 1 (dashboard-homepage) | prod (self-monitor) |

OBSERVED. **Backend / third-party hosts** that appear ONLY inside network-anchor predicates
(never navigated to):

| # | host | where | nature |
|---|---|---|---|
| 6 | `wegapi.azure-api.net` | browse-menu:30, catering-browse:41, meals2go-homepage:38 | first-party API on Azure APIM (multi-tenant PaaS domain) |
| 7 | `*.algolia.net` | search-autocomplete:27 | third-party search SaaS (multi-tenant) |
| 8 | `www.opentable.com` | amore-reservations:26, nextdoor-reservations:32 | third-party reservations widget |

### 2b. Could a host→environment map be derived unambiguously?

**Partially, and only for prod.** All five navigation hosts are prod; **no staging/dev host
exists anywhere in the repo**, so a derived map has an empty staging/dev side and cannot be
validated against a non-prod exemplar. That itself is the S2 caveat: the flagship
pre-prod-regression arc needs non-prod hosts that this fleet does not yet contain.

**Hosts a naive regex map would MISCLASSIFY or leave UNMATCHED (→ must be `environment:unknown`,
not defaulted to prod):**

1. **`synthwatch-dashboard.vercel.app`** — `*.vercel.app` is Vercel's **shared deploy
   domain, used for BOTH preview and production deployments**; the hostname alone does not
   disclose the environment. It is also a *different property* (the SynthWatch self-monitor),
   not a Wegmans-family origin. A map keyed on the Wegmans host families leaves it unmatched.
   → **unknown**, never assume prod.

2. **`wegmansamore.com`** (no `www.`) — every other Wegmans-family host is `www.`-prefixed,
   **but its sibling `www.wegmansnextdoor.com` IS `www.`-prefixed**. The family is internally
   inconsistent on the prefix, so any regex that assumes uniformity (`^www\.wegmans…`) drops
   `wegmansamore.com`. → must be matched by exact host, not by a `www.`-anchored pattern, or
   it renders unmatched.

3. **The backend/third-party hosts (#6–#8)** — `wegapi.azure-api.net` and `*.algolia.net`
   are **multi-tenant PaaS/SaaS domains** and `www.opentable.com` is an **unrelated third
   party**. If a naive map ingests *every* host it sees in a spec (including anchor hosts, not
   just `goto` targets), these get bucketed against a Wegmans environment they don't belong
   to. None is environment-bearing for Wegmans; they must be **excluded or `unknown`**, never
   prod. (`azure-api.net` in particular would tempt an "Azure ⇒ prod-infra" heuristic that is
   simply wrong for env classification.)

**Unambiguous (exact-host keying):** `www.wegmans.com`, `www.meals2go.com`,
`www.wegmansnextdoor.com` map cleanly to prod.

**Caveat / falsifier (run):** falsifier was "a non-prod host exists somewhere" — the full
`grep -rniE 'https?://' monitors/` census (§1b table + anchor consts) shows only prod
origins; no `staging`/`dev`/`qa`/`.test`/preview host appears. So the map's non-prod side is
unpopulated by construction, and the three shapes above are the ones a regex must refuse to
guess on.

### 4b. What that (unmerged) §6 documents — summary

OBSERVED (read from the branch copy, `…:docs/analysis/2026-07-deep-review.md` lines 281–341).
Its §6 is titled **"BOUNDARY CONTRACTS"** and records *the format this repo exposes to the
SynthWatch runner*, in two sub-contracts:

1. **The manifest contract** (`manifest.json` / `manifest.schema.json`) — a field-by-field
   table (`id`, `name`, `script`, `kind`, `sensitive`, `redact_patterns`,

   cross-checked against the URL the spec drives"** (this directly corroborates §1c above).

2. **The spec contract** (what the runner assumes about script shape) — a spec is a standard
   `@playwright/test` file; each `test.step` maps to a runner `run_step` funnel label; imports
   from `../../lib/flow` are **esbuild-aliased at runtime to the runner's vendored

   budget**.

**This is the "what this repo consumes from the runner shim + api" boundary** the T5.2 thread
wanted: (a) the manifest schema the runner reads, and (b) the `lib/flow` → `specShim` vendor
block + esbuild alias + parity-SHA coupling.

**OBSERVED — the §6 doc's headline gap is now STALE.** Since it was written, `main` gained

even as the shim evolves. Recommend (out of scope to change here) that the CI treat a fetch
failure as a **hard error** — or at least surface it beyond a `::warning::` — so a moved file
