## 1. ★S1/S2 — Target-host parameterizability

**ANSWER (ground truth): NOT cleanly parameterizable.** For a browser monitor the target
host is **baked into the flow code** in the spec body (the `page.goto(...)` literal), not a
declarative field. The manifest carries a `target` field but **nothing reads it** — it is
documentation, not configuration. There is **no** endpoint/HTTP monitor to compare against;
that monitor class does not exist in this repo and is not even representable in the schema.

### 1a. There is no ENDPOINT/HTTP monitor (the second half is vacuous)

OBSERVED — every manifest entry is `kind: "browser"` (16/16), and the schema's `kind` is a
single-value enum:

```
manifest.schema.json:32   "kind": { "type": "string", "enum": ["browser"] },
$ grep -c '"kind": "browser"' manifest.json  → 16   (every monitor)
```

So "a representative ENDPOINT/HTTP monitor" cannot be shown — the fleet is 100% browser
journeys. Any HTTP-check parameterizability question is answered only by whatever the
runner does for non-browser checks, which is **out of this repo** (INFERRED-unknown).

### 1b. The browser host is hardcoded per-spec in `page.goto()`

OBSERVED — every spec navigates to an absolute, hardcoded origin. Representative
(`monitors/wegmans/homepage-load.spec.ts:21`):

```ts
await page.goto('https://www.wegmans.com', { waitUntil: 'domcontentloaded' });
```

### 1c. The manifest `target` field is read by nothing

OBSERVED — `target` exists per-entry (e.g. `manifest.json:14 "target": "https://www.wegmans.com"`),
but:

- No spec references it. `grep -rniE 'process\.env|baseURL|import\.meta|env\.' monitors/`
  → **(none)**.
- `playwright.config.ts` sets **no `baseURL`** (so a relative `goto` has nothing to resolve
  against; the absolute literals are load-bearing). Config header itself says the runner
  "applies its OWN execution config" (`playwright.config.ts:5-11`).
- The manifest validator never cross-checks `target` against the spec's `goto` — corroborated
  by the (unmerged) prior deep-review §6: *"`target` … validated by **nothing** — not even
  cross-checked against the URL the spec drives"* (see §4).

So `target` is decorative: changing it would repoint **nothing**; the browser still goes
wherever the `goto` literal says.

### 1d. Verdict + lift

**Can the same artifact be pointed at a different environment's host without editing flow
code — via a spec field, env var, or runtime override? → NO.** There is no such seam today:
no `baseURL`, no `process.env`, no consumption of `target`. Repointing requires **editing the
spec source**.

Lift to make the fleet env-repointable (OBSERVED counts of host literals in *executable*
code, comments excluded):

- **16 navigation-host literals** — one `goto` (or `ENTRY_URL`) per spec, all 16 specs.
- **6 additional backend/third-party host literals** in module-level network-anchor
  regexes that also pin a host:
  - `wegapi.azure-api.net` — `meals2go-browse-menu.spec.ts:30` (`MENUS_API`),
    `meals2go-catering-browse.spec.ts:41` (`CATERING_MENUS_API`),
    `meals2go-homepage.spec.ts:38` (`BOOTSTRAP_API`)
  - `algolia.net` — `search-autocomplete.spec.ts:27` (`ALGOLIA_QUERIES`)
  - `opentable.com` — `amore-reservations.spec.ts:26` + `nextdoor-reservations.spec.ts:32`
    (`OPENTABLE_LOADER`; a fixed third party — would NOT change per Wegmans environment)

Net edit surface for a full env repoint: **~22 host literals across 16 files**, none behind a
shared declarative field. A first cut (nav hosts only) is 16 edits in 16 files; the
network-gated specs additionally need their anchor host swapped or they would gate on the
wrong environment's backend.

**Caveat / falsifier (run):** falsifier was "some spec reads an env var or `target`" —
`grep -rniE 'process\.env|baseURL|import\.meta|env\.' monitors/` returned nothing, and
`playwright.config.ts` has no `baseURL`. The seam genuinely does not exist. INFERRED-only
piece: how the *runner* injects config is out of this repo — if the runner string-replaced
hosts at sync time it could parameterize externally, but nothing in-repo exposes or expects
that.
