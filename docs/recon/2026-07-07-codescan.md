## STEP 0 — Is code scanning even ON here?

**ANSWER (ground truth): NO. CodeQL / GitHub code scanning is NOT enabled on
`craigoley/synthwatch-monitors`.** There is no analysis, no default setup, and no CodeQL
workflow. So there is **no backlog to work** — the "execute the fixable / propose the rest"

### 0a. The alerts endpoint returns 404 "no analysis found" (not 403)

OBSERVED:

```
$ gh api "repos/craigoley/synthwatch-monitors/code-scanning/alerts?state=open" --jq 'length'
  → {"message":"no analysis found", … ,"status":"404"}   (HTTP 404)
$ gh api "repos/craigoley/synthwatch-monitors/code-scanning/analyses"
  → {"message":"no analysis found", … ,"status":"404"}   (HTTP 404)
```

A **404 "no analysis found"** is the *scanning-not-enabled* signal. A token that lacked the
`security_events` scope would return **403**, not 404 — so this is genuinely "nothing has ever
been scanned," not a permission artifact. (`gh` appends a generic `admin:repo_hook` scope hint
to these 404s; that hint is misleading here — see 0b, which disproves a scope problem.)

### 0b. Default setup is `not-configured` — and this 200 proves it's not a token-scope issue

OBSERVED:

```
$ gh api "repos/craigoley/synthwatch-monitors/code-scanning/default-setup"
  {"state":"not-configured",
   "languages":["actions","javascript","javascript-typescript","typescript"],
   "query_suite":"default","threat_model":"remote","updated_at":null,
   "schedule":null,"runner_type":"standard","runner_label":null}
```

This call returned **HTTP 200** with a real body. That is decisive on two points:

1. **The token can read code-scanning configuration** — a missing-scope token would 403 this
   endpoint too. It didn't. So 0a's 404s are "no data," not "no access." (This is the
   falsifier for "maybe scanning IS on but I just can't see it" — run, and it failed: the
   config is readable and says not-configured.)
2. **CodeQL default setup is explicitly `not-configured`** (`updated_at: null`, `schedule:
   null`) — it was never turned on. GitHub *could* scan the languages it detected
   (`javascript-typescript`, `actions`), but nothing is scheduled or advanced-configured.

## PROPOSAL (Craig's decision — enable CodeQL or not)

**Recommendation: LOW priority / optional. Enable CodeQL default setup only if the near-zero
maintenance cost is acceptable; otherwise the hand-rolled gates are a defensible posture for a
repo of this shape.** One-line tradeoff:

> **CodeQL on a Playwright-spec repo is low-yield** — the specs are short, trusted, reviewed
> browser journeys with no user input, no server, no dynamic `eval`, and a fixed vetted
> dependency set (`README.md` security model) — **but it would catch injection/regex classes
> the #44/#46 hand-rolled gates structurally cannot**, most plausibly `js/regex-injection` or
> `js/incomplete-sanitization` around the network-anchor `RegExp`s and any future
> `new RegExp(userValue)` in a spec or in `redact_patterns` handling.

Considerations, evidenced:

- **Yield is low but not zero.** The repo already hand-builds `RegExp`s
  (`check-matchers.mjs` parses `SUPPORTED_MATCHERS`; `validate-manifest.mjs` compiles each
  `redact_patterns` string via `new RegExp(p)`). Those are the exact shapes CodeQL's regex/
  injection queries target. A `scratch/`-style probe (gitignored, per CLAUDE.md) could also
  trip queries — but `scratch/` never reaches `main`, so default setup (which scans the
  default branch) wouldn't see it.
- **Cost is near-zero.** GitHub already detected the languages (0b). Default setup is a
  toggle; no workflow to maintain. It would add one required check to the PR gate you already
  merge behind.
- **Not unilateral.** Enabling changes the repo's security surface + PR-gate — a Craig
  decision, so this is a proposal, not an action taken.

No action. This ledger records the deliberate posture: **monitors is gated by #44 (ajv
